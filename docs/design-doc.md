## Problem Statement

Vet students at the Faculty of Veterinary Medicine, University of Jos (200–600 level) have no structured, centralized place to access past questions and study materials. Resources are scattered across WhatsApp groups, individual drives, and informal sharing. The platform Nel VetVault already exists at nelvetvault.com.ng but is on paid hosting (DomainKing) that the builder wants to eliminate. The rebuild goal: a React SPA hosted free on Netlify, backed by Supabase, with community-driven uploads, admin vetting, and a clear path to AI features, quiz mode, and peer study groups.

## What Makes This Cool

The hook is: **students are contributors, not just consumers.** Every vet student who uploads a past question makes the platform more valuable for the next. The platform grows itself. Combined with radical specificity — this is *for Unijos vet students specifically*, organized by course code and level, not a generic Nigerian past-question dumping ground — that specificity is the moat.

Landscape gap: ExamPastQuestions, PassMark, SchoolNGR, and Ulearngo all exist, but they cover JAMB/WAEC/NECO nationally. None are faculty-specific, course-code-organized, or community-upload-driven for a single university department. NelVetVault is the first mover in this vertical.

## Constraints

- Free hosting only (Netlify for frontend, Supabase free tier for backend)
- Builder is learning React — architecture should be learnable, not just functional
- Platform starts open (no login to browse or upload) — auth comes later
- Admin vetting: all uploads go live only after admin approval
- Mobile-first — Nigerian students are primarily mobile users
- PDF is the primary file format

## Premises (All Agreed)

1. Platform starts open — no login required to browse or upload
2. Admin vetting: materials are pending until an admin explicitly approves them
3. A real backend is needed — file storage + database for metadata + approval state
4. React SPA on Netlify + Supabase for everything backend
5. MVP scope: browse materials + upload a file + admin approves it

## Recommended Approach: React + Supabase

**Tech stack:**
- **Frontend:** React (Vite), React Router, Tailwind CSS
- **Backend:** Supabase (Postgres + Storage + Auth)
- **Hosting:** Netlify (frontend), Supabase cloud (backend)
- **Version control:** GitHub → Netlify auto-deploy on push

**Database schema (core tables):**
```
materials
  id, title, course_code, level (200/300/400/500/600),
  type (past_question | study_material),
  year (nullable, 4-digit integer), file_url, file_size,
  status (pending | approved | rejected),
  uploaded_at, approved_at

courses (optional enrichment table — skip for MVP, add later)
  code, name, level, description
```
Note: `admin_notes` removed from MVP schema — adds UI complexity not needed yet.

**Security / Row Level Security (RLS):**
All Supabase tables need RLS policies set up immediately — without them, any user can read/write everything via the auto-generated API.
- `materials` SELECT: `WHERE status = 'approved'` (public reads approved only)
- `materials` INSERT: allow for all (anonymous upload is intentional)
- `materials` UPDATE: restrict to authenticated admin (status changes only)
- Storage bucket: public-read (for download links), write-restricted (uploads go via service role in a Supabase Edge Function or via anon key with bucket policy limiting MIME type to `application/pdf`)
- Storage MIME restriction: configure bucket to accept PDF only in Supabase dashboard → Storage → Bucket settings. Note: the dashboard MIME toggle is a UX guard, not a hard server-side boundary — direct API calls can bypass it. For MVP this is acceptable. If non-PDF uploads appear in the bucket, add a Supabase Edge Function to validate `content-type` server-side before writing.
- Note: Free tier gives 1GB storage — monitor manually via Supabase dashboard

**Environment variables (critical for learners):**
```
# .env at project root (add .env to .gitignore immediately)
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-anon-key-here
```
These same values must be added in Netlify → Site Settings → Environment Variables before deploying.

**Key pages/routes:**
- `/` — Homepage: search bar, level filter pills, recent approved materials grid
- `/browse` — Full browse with filters (level, type, course code, year)
- `/upload` — Upload form: title, course code, level, type, year, file picker
- `/material/:id` — Material detail page: shows title, course code, level, type, year, file size, upload date, and a "Download" button that opens the Supabase Storage public URL in a new tab. No authentication required.
- `/login` — Admin login page (email + password via Supabase auth). Redirect to /admin on success.
- `/admin` — Protected: if no active session, redirect to /login. If authenticated, show pending uploads list with approve/reject buttons per row.

**Upload form validation:**
- `title`: required, 5–120 characters
- `course_code`: required, free text for MVP (e.g. "VPH 404"), 2–20 characters uppercase recommended
- `level`: required, select from 200 / 300 / 400 / 500 / 600
- `type`: required, select from "Past Question" / "Study Material"
- `year`: optional, 4-digit integer (1990–current year)
- `file`: required, PDF only (validate MIME type client-side before upload), max 50MB

**Upload flow:**
1. Student fills form and selects PDF → client validates MIME type and size
2. File uploads to Supabase Storage → on success, insert row with `status: pending`
3. If Storage upload fails → show error message, do NOT insert DB row
4. If DB insert fails after Storage upload → delete the orphaned file from Storage
5. On success → show confirmation: "Your file is under review. It will appear once approved."
6. Admin navigates to /admin → sees pending list → clicks Approve or Reject → status updates
7. Material appears in public browse only after status = 'approved'

**UI design direction:**
- Colors: dark green (#1a4a2e) primary, gold (#c9a84c) accent — evokes university, trust, professionalism
- Mobile-first grid layout
- Search is prominent on hero
- Level filter pills (200-600) — single click to filter
- Material cards: course code badge, type tag (Past Question / Study Material), year, download button
- Upload CTA section on homepage — dashed gold border, inviting

**Wireframe:** A layout sketch was generated during the office hours session. Layout: dark green nav with gold "VetVault" accent and "+ Upload" CTA button → hero section with search bar → level filter pills (All / 200-600) → material cards grid (course code badge, type tag, year, download link) → dashed gold upload CTA banner → admin panel shows pending queue with approve/reject per row.

## Open Questions

1. What course codes exist in the Faculty of Vet Med at Unijos? Should we seed a `courses` table or let uploaders enter them freehand?
2. File size limit for uploads? Supabase Storage free tier allows up to 50MB per file — is that enough for PDFs?
3. Admin identity: is there one admin (you), or multiple? Supabase auth can handle multiple admins via role column.
4. Do you want uploaders to optionally enter their name, or keep anonymous forever?
5. Domain plan: nelvetvault.com.ng points to DomainKing now — will you redirect it to Netlify, or use the free `.netlify.app` subdomain for now?

## Success Criteria

- A vet student can browse materials without logging in
- Upload form is functional and a test PDF can be submitted end-to-end in a single session (network speed dependent)
- Admin can see all pending uploads and approve/reject in one click
- Platform is live on Netlify with a working Supabase connection
- Git repo on GitHub, Netlify auto-deploys on push to main
- At least one real material is browsable on launch day

## Distribution Plan

- Frontend: Netlify (free tier, auto-deploy from GitHub)
- Backend: Supabase cloud (free tier — 500MB DB, 1GB Storage, 50MB/file)
- Domain: Either `nelvetvault.netlify.app` (free) or point existing `nelvetvault.com.ng` to Netlify via DNS change at DomainKing
- CI/CD: GitHub push to `main` triggers Netlify build — zero extra config
- Note: Supabase free tier pauses projects after 7 days of inactivity. For MVP, manually unpause via Supabase dashboard as needed. Upgrade to a paid plan if traffic picks up.

## Next Steps (Build Order)

1. `git init nelvetvault && cd nelvetvault` — init git repo first
2. `npm create vite@latest . -- --template react` — scaffold React app into the repo
3. Create `.env` file with `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`. Add `.env` to `.gitignore` immediately.
4. Create Supabase project at supabase.com → create `materials` table → enable Storage → configure bucket for PDF-only MIME → set up RLS policies (see Security section above)
5. Push to GitHub: `git add . && git commit -m "init" && git remote add origin <your-repo> && git push -u origin main`
6. Connect Netlify to GitHub repo — Netlify auto-detects Vite, add env vars in Netlify dashboard
7. Build homepage — static layout first (Tailwind, green/gold colors, no real data)
8. Wire up Supabase client — install `@supabase/supabase-js`, fetch approved materials, display cards
9. Build `/upload` route — form with validation, Supabase Storage upload, DB insert, error handling (on Storage failure: show error, no DB row; on DB failure after upload: call `supabase.storage.from('materials').remove([filePath])` — if that fails, log it and note the orphan for manual cleanup in the Supabase dashboard)
10. Build `/material/:id` route — fetch single material by id from Supabase, display title/course/level/type/year/file size, render Download button pointing to the Storage public URL
11. Build `/login` + `/admin` routes — Supabase auth, redirect logic, pending materials queue, approve/reject
11. Verify full flow end-to-end: upload → pending → admin approves → appears in browse
